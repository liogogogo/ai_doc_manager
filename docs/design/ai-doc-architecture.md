# AI 自动化流程下的文档架构管理指南

> **版本**: 1.1 · **日期**: 2026-02-28
> **背景**: 在 AI 深度参与编码的自动化流程下，传统的文档管理方式（如庞大复杂的 Wiki 树、说教式的长篇规范）已成为 AI 产生幻觉和注意力稀释的重灾区。本文档基于第一性原理与前沿最佳实践，重新定义 AI 时代的文档架构管理。

---

## 一、第一性原理：AI 时代的文档悖论

在人类主导的时代，文档是**为人类大脑的检索而设计的**（按模块分类、按时间线铺陈、充满背景故事）。
但在 AI 主导的自动化流程下，文档变成了**Prompt 的一部分（上下文）**。

这带来了一个核心悖论：**文档越详尽，AI 的输出质量往往越差。**

### AI 阅读文档的物理限制
1. **注意力稀释 (Attention Dilution)**：在 100K Token 的上下文中，核心约束被淹没在冗余叙事中。
2. **知识污染 (Knowledge Pollution)**：过时的旧文档和新版架构设计同时存在，会导致 AI 陷入逻辑冲突。
3. **行为取代 (Behavior Replacement)**：人类通过读文档学习“流程”，而 AI 需要的是“可执行的门禁脚本（Hooks/Guard）”。

**结论**：在 AI 工作流中，文档不再是“知识库”，而是**“强类型的上下文契约（Typed Context Contract）”**。

---

## 二、文档架构重塑：极简四层模型

基于上述认知，一个适应 AI 工作流的理想文档架构应该被极其严格地划分为以下四个层级：

### 1. 宪法与边界层 (Rule & Boundary)
- **特征**：极简、高频加载、硬性约束。
- **内容**：`AGENTS.md` (或 `.cursorrules`)、`AI_GOVERNANCE_CONSTITUTION.md`。
- **原则**：**只有“绝对不能做的事”才有资格进入这一层。** 任何可以被 linter 或 guard 脚本自动检查的规则（如“必须格式化代码”），都应从文档中删除，转为代码实现。

### 2. 状态与记忆层 (State & Memory)
- **特征**：动态、读写频繁、需定期垃圾回收（GC）。
- **内容**：`.ai/progress.md`、`.ai/archive/`。
- **原则**：这是 AI 的“短期工作记忆”。必须设定容量上限（如 100 行）。一旦超过，必须强制将已完成的事项归档（Archive），以对抗注意力稀释。

### 3. 契约与蓝图层 (Contract & Blueprint)
- **特征**：结构化、业务强相关、单点真相（SSOT）。
- **内容**：`docs/design/*.md` (如 `jwt-auth.md`, `im-api-spec.md`)、Proto 文件。
- **原则**：这是 AI 编码的**输入契约**。这里不需要写“如何实现”，只需写明“验收标准（Acceptance Criteria）”、“输入输出数据结构”和“关键业务流”。

### 4. 决策快照层 (Decision Snapshot)
- **特征**：只读、防回溯、解决“为什么”。
- **内容**：`docs/adr/*.md` (架构决策记录)。
- **原则**：主要给人类看，或者当 AI 质疑某个反直觉的设计时提供背景支撑（例如：为什么我们不用标准库而要引入一个特定的第三方包）。

## 三、文档管理的“反模式”与清理策略

如果你的项目中有以下文档，在 AI 工作流下它们往往是“有毒资产”，建议立即清理或降级：

| 反模式文档类型 | 为什么在 AI 时代有毒？ | 处理策略 |
| :--- | :--- | :--- |
| **工作流/流程指南** (如“开发 SOP”) | AI 不需要人类的 Step-by-Step 说教。 | **删除**。转为可执行的 IDE Workflows 或 Shell 脚本。 |
| **技术雷达/技术栈罗列** | 这些应内化在系统提示词或环境配置中。 | **删除**。将必须使用的框架合并到“宪法”的一句话约束中。 |
| **过时的架构图 / V1 版设计** | 极易造成知识污染，让 AI 错误调用废弃接口。 | **强制归档或加前缀标记**（如 `[DEPRECATED]`）。 |
| **长篇的“如何做”教程** | AI 是通过阅读源码和具体示例（Few-shot）学习的，而非阅读自然语言。 | **重写为代码示例** (Golden Examples) 并放在 `best_practices/` 中。 |

## 四、关于“开发专门的文档管理智能体”的可行性评估

**核心观点：这是一个极高收益、且直击当前 AI 研发痛点的绝佳方向。**

当前所有的 AI 编码助手（Windsurf, Cursor, Claude Code）都在致力于“生成代码”，但**没有一个工具在致力于“维护上下文的纯净度”**。

在自动化流程中，“文档腐化”是比“代码腐化”更可怕的暗物质。一个专门的文档管理智能体（Doc-Guardian Agent）将填补这一空白。

### 预期的核心能力（Use Cases）

1. **记忆垃圾回收器 (Memory GC)**
   - **痛点**：`progress.md` 无限膨胀。
   - **功能**：智能体定期分析 `progress.md`，自动将已完结的 Epic/Milestone 提取重写为精简的摘要，并移动到 `archive/` 目录，始终保持工作记忆在 100 行以内。

2. **知识污染检测 (Knowledge Conflict Detector)**
   - **痛点**：修改了代码逻辑，但 `docs/design/` 下的旧文档没更新，下次 AI 又按旧文档写出了 Bug。
   - **功能**：当 PR 涉及核心架构变更时，智能体通过向量检索比对 `docs/` 下的所有文件，找出与新代码逻辑冲突的自然语言描述，并自动提交针对这些文档的修正 PR。

3. **隐式规则提取 (Implicit Rule Extraction)**
   - **痛点**：人类在 Code Review 中反复纠正 AI 同一个错误，但没人去更新 `AGENTS.md`。
   - **功能**：监控 `.ai/failures.jsonl` 或 Git 评论，当发现同一类违规出现 ≥3 次时，智能体自动弹窗提议：“我发现我们经常在 Redis 锁上犯错，建议将这段正确的代码作为 Golden Example 加入最佳实践库中，是否执行？”

4. **冗余杀手 (Redundancy Pruner)**
   - **痛点**：项目中堆积了大量描述性废话。
   - **功能**：扫描代码库，识别出那些试图用自然语言描述“可以通过 linter 或脚本硬约束”的文档段落，并建议将其删除，替换为具体的 `make guard` 规则。

### 评估结论
- **技术可行性**：高。基于现有的 RAG、语义比对和工具调用能力完全可实现。
- **业务收益**：极高。它解决的是目前 AI 研发效率的**瓶颈制约**——上下文质量问题。管理好文档（上下文），就是管理好了 AI 的“智商”。

## 五、DocGuardian 实现现状（v0.5）

> 以下为 DocGuardian 项目当前已实现的核心能力，与上述理论架构的对应关系。

### 5.1 已实现核心功能

| 功能 | 对应理论 | 实现方式 |
|------|---------|---------|
| **F0: Project Initializer** | 宪法与边界层自动生成 | 确定性扫描（tech_detector.rs）→ 规则引擎生成基础 AGENTS.md |
| **F5: LLM 智能生成引擎** | 宪法层的智能化升级 | 增强扫描（依赖树+CI）注入 prompt → LLM 生成 12 章节 AGENTS.md → 流式输出 + 多轮优化 + 选中定向优化 |
| **F6: 偏差检测** | 治理闭环 | 确定性扫描 vs AGENTS.md 内容对比 → 偏差报告 → LLM 增量更新建议 |

### 5.2 12 章节治理体系

DocGuardian 生成的 AGENTS.md 采用前沿的 12 章节结构，覆盖 AI 编码治理的完整生命周期：

```
1. 头部元信息      — 版本号 + 更新日期 + 项目描述 + 适用 AI 助手
2. Scope（权限边界） — 白名单/只读/黑名单，粒度到目录级
3. Don't（安全红线） — 违反即停的硬性约束
4. Ask First（需确认）— 新增依赖/改 API/删接口等需人工确认
5. Commands（验证命令）— 三级验证：文件级(秒级) / 模块级(分钟) / 提交前(全量)
6. Project Structure  — 树形图 + 权限注解（✏️可写/👁只读/🚫禁止）
7. Verification       — AI 完成任务后的必执行清单
8. Memory             — 会话记忆管理（读/写 .ai/progress.md）
9. Examples           — ✅ Good / ❌ Bad 对比，匹配项目技术栈
10. When Stuck        — 搜代码确认 + 查文档 + 熔断机制
11. Governance Loop   — 治理框架自身的更新机制（闭环）
12. Changelog         — 版本化更新记录
```

### 5.3 增强扫描深度（v0.5 新增）

传统的项目扫描仅检测文件存在性。DocGuardian v0.5 引入**依赖树深度解析**，使 LLM 能基于具体依赖生成针对性治理规则：

```
扫描数据注入 prompt 的效果对比：

注入前（仅文件检测）:
  "项目使用 Rust + TypeScript, 有 Tauri 框架"
  → LLM 生成通用规则

注入后（依赖树 + CI 配置）:
  "npm 依赖: react@^18.3, zustand@^5.0, rusqlite@0.32
   CI: GitHub Actions [build: push/pull_request, steps=lint→test→build]"
  → LLM 生成：
    - 针对 rusqlite 的 SQL 注入防护规则
    - 针对 zustand 的状态管理约束
    - 匹配实际 CI 步骤的验证命令
```

### 5.4 治理闭环架构

```
┌─────────────────────────────────────────────────────┐
│                  治理闭环 (Governance Loop)            │
│                                                     │
│  项目演进 ──→ 增强扫描 ──→ 偏差检测 (F6)             │
│                              │                      │
│                              ▼                      │
│                         偏差报告                     │
│                     (语言/框架/工具/                  │
│                      目录/章节/CI/测试)              │
│                              │                      │
│                              ▼                      │
│                    LLM 增量更新建议                   │
│                              │                      │
│                              ▼                      │
│                    人工审核确认                       │
│                              │                      │
│                              ▼                      │
│                    写入 AGENTS.md                    │
│                    (版本号 + Changelog)              │
│                              │                      │
│                              ▼                      │
│                    Git 版本管理                       │
│                              │                      │
│                              └──→ 项目演进（循环）    │
└─────────────────────────────────────────────────────┘
```

这一闭环确保治理框架与项目始终同步，避免"写了一次再也不更新"的常见失效模式。
